Crawlix Next is a queue-based scraping platform built with:

- `Next.js` for the internal dashboard
- `NestJS` for the API service
- `NestJS` for the worker service
- `RabbitMQ` for job queueing, retry, and dead-letter routing
- `Redis` for cache, job state, result storage, and worker heartbeat

This documentation is authored with Fumadocs using content-driven `.mdx` pages inside `content/docs`.

## Platform Summary

Crawlix separates request handling from execution:

1. The API receives a request and enqueues a scrape job.
2. RabbitMQ distributes the job to available workers.
3. Workers run scraping strategies and write status and result data to Redis.
4. The dashboard reads system state from the API for operations and monitoring.

## Public vs Admin Surface

The docs intentionally separate public and admin behavior:

- Public/programmatic API endpoints include request examples.
- Admin-only endpoints such as dashboard login, API key management, retry, and cancel are documented conceptually but do not include request examples here.
- Dashboard interactions are intended for internal operators using session-based authentication.
