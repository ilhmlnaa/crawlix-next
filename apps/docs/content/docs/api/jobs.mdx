## POST /api/jobs

Creates a new scrape job.

### Auth

Requires API key or authenticated admin session.

### Example Request

```bash
curl -X POST http://localhost:3001/api/jobs \
  -H "Content-Type: application/json" \
  -H "x-api-key: cx_your_api_key_here" \
  -d '{
    "url": "https://example.com",
    "strategy": "auto",
    "options": {
      "timeoutMs": 30000,
      "useCache": true,
      "cacheTtlSeconds": 900,
      "maxRetries": 2,
      "retryDelayMs": 1000,
      "waitUntil": "domcontentloaded"
    }
  }'
```

### Example Request With Targeted Worker

Use `targetWorkerId` when you want the job to run only on one specific worker instance.

```bash
curl -X POST http://localhost:3001/api/jobs \
  -H "Content-Type: application/json" \
  -H "x-api-key: cx_your_api_key_here" \
  -d '{
    "url": "https://example.com/protected",
    "strategy": "playwright",
    "targetWorkerId": "worker-host123-4567",
    "options": {
      "timeoutMs": 45000,
      "useCache": false
    }
  }'
```

### Example Request With Webhook and Idempotency

Use `webhookUrl` when you want Crawlix Next to push a terminal job event to your server, and provide `Idempotency-Key` to make client retries safe.

```bash
curl -X POST http://localhost:3001/api/jobs \
  -H "Content-Type: application/json" \
  -H "x-api-key: cx_your_api_key_here" \
  -H "Idempotency-Key: job-request-001" \
  -d '{
    "url": "https://example.com",
    "strategy": "auto",
    "webhookUrl": "https://your-app.example/webhooks/crawlix",
    "options": {
      "timeoutMs": 30000,
      "useCache": true
    }
  }'
```

### Example Response

```json
{
  "jobId": "job_01hrx8qj2b1j6m3n9t1",
  "status": "queued",
  "progress": 0,
  "stage": "queued",
  "queuedAt": "2026-04-11T07:10:25.000Z",
  "resultTtlSeconds": 3600,
  "targetWorkerId": "worker-host123-4567",
  "webhookUrl": "https://your-app.example/webhooks/crawlix",
  "idempotencyKey": "job-request-001"
}
```

## Request Body

```json
{
  "url": "https://example.com",
  "strategy": "auto",
  "targetWorkerId": "worker-host123-4567",
  "webhookUrl": "https://your-app.example/webhooks/crawlix",
  "webhookSecret": "optional-per-job-secret",
  "idempotencyKey": "job-request-001",
  "options": {
    "timeoutMs": 30000,
    "method": "GET",
    "body": "optional raw body",
    "headers": {
      "User-Agent": "Crawlix Client"
    },
    "formData": {
      "key": "value"
    },
    "useCache": true,
    "cacheTtlSeconds": 900,
    "maxRetries": 2,
    "retryDelayMs": 1000,
    "waitUntil": "domcontentloaded",
    "waitForSelector": "#app",
    "waitForFunction": "window.__READY__ === true",
    "additionalDelayMs": 500,
    "useProxy": false,
    "proxyUrl": "http://206.189.154.204:8888"
  }
}
```

## Proxy Options

If you want proxy routing per job, use `options.useProxy` and optionally `options.proxyUrl`.

```json
{
  "url": "https://example.com/protected",
  "strategy": "playwright",
  "options": {
    "useProxy": true,
    "proxyUrl": "http://206.189.154.204:8888"
  }
}
```

## Notes

- `strategy` supports `auto`, `cloudscraper`, and `playwright`.
- `auto` is the recommended default.
- `targetWorkerId` is optional. Omit it to let Crawlix Next auto-assign the job through the shared queue.
- `webhookUrl` is optional. When present, Crawlix Next will push terminal job events like `job.completed` and `job.failed` to that endpoint.
- `webhookSecret` is optional. If omitted, the worker falls back to the global `WEBHOOK_SIGNING_SECRET`.
- `idempotencyKey` can be supplied in the request body or the `Idempotency-Key` header.
- Job status now includes `progress` and `stage` so polling clients can show richer job lifecycle updates.
- If `targetWorkerId` is provided, the API validates that the worker is still active before accepting the job.
- Targeted jobs keep their worker affinity during retry and are dispatched to the selected worker queue, not the shared queue.
- Status polling is the default client pattern in v1.
- Webhook payloads are signed using HMAC SHA-256 headers:
  - `x-crawlix-event`
  - `x-crawlix-event-id`
  - `x-crawlix-timestamp`
  - `x-crawlix-signature`
- Admin-only endpoints like retry and cancel are intentionally omitted from request examples in this docs section.
- Proxy decision order:
  1. if `options.useProxy` is `false`, Crawlix does not use a proxy.
  2. if `options.useProxy` is `true` and `options.proxyUrl` is present, worker uses `options.proxyUrl`.
  3. if `options.useProxy` is `true` and `options.proxyUrl` is empty, worker falls back to `SCRAPER_PROXY_URL` from worker env.
  4. if `options.useProxy` is `true` and both values are empty, request runs without proxy.
